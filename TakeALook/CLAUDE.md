# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Unity 6 (`6000.2.10f1`) first-person shooter / horror game. Uses URP, the new Input System, TextMeshPro, DOTween (`Assets/Plugins/Demigiant/DOTween`), and PSXShaderKit for the retro PS1 look. Code comments are mostly Spanish; identifiers are English. There is no automated test/build CLI in this repo — building, playing, and testing happen inside the Unity Editor.

## Working in this repo

- Open the project by pointing Unity Hub at the repo root (the folder containing `Assets/`, `ProjectSettings/`, `Packages/`). Use Unity `6000.2.10f1` exactly — opening with a different version will rewrite project files.
- Game code lives in `Assets/_TakeALook/`. Other top-level `Assets/` folders (`Cosmic_Retro_*`, `PSXShaderKit`, `_ParticlesFLO_A`) are third-party asset packs — treat them as vendored content, don't refactor them.
- `.csproj` / `.slnx` files are auto-generated by Unity; never edit by hand. Adding a new `.cs` file under `Assets/` is enough — Unity regenerates the project on focus.
- `TakeALook/Assets/_TakeALook/Scripts/Enemys/HeadshotCollider.cs` is a stray duplicate at the repo root (outside the Unity project's `Assets/`). The canonical version lives at `Assets/_TakeALook/Scripts/Enemys/HeadshotCollider.cs`. Don't edit the stray; flag it if it gets in the way.
- Scenes live in `Assets/_TakeALook/Scenes/`. `SCN_LVL1.unity` / `SCN_LVL2.unity` / `SCN_LVL3.unity` are the gameplay levels; `SCN_Menu.unity` is the entry point. Several `*_Test_*.unity` and `Prop_Test.unity` scenes are scratch — don't assume they reflect real gameplay.

## Architecture

### Singleton managers (DontDestroyOnLoad)

Three persistent singletons drive cross-scene state. They follow the same `Instance` pattern with self-destruction on duplicate `Awake()`:

- **`GameManager`** (`Assets/_TakeALook/Scripts/Managers/GameManager.cs`) — source of truth for run-level persistence. Tracks picked-up item IDs and killed-enemy IDs as `HashSet<string>` (rebuilt on Awake from serialized `List<string>`), so items/enemies don't respawn when the player re-enters a scene. Also owns `runTime` and `currentRoomName`. Exposes `OnRoomChanged` and `OnInventoryChanged` events. **When adding any spawnable that should not respawn after death/pickup, give it a stable GUID and register through `GameManager`.** Call `ResetRunState()` on new game.
- **`AudioManager`** (`Assets/_TakeALook/Scripts/Managers/AudioManager.cs`) — pooled `AudioSource`s (separate SFX, UI, music groups), looked up by string ID against a `SoundLibrary` ScriptableObject. Code references sounds by ID strings (e.g. `"gun_draw"`, `"player_step"`); the actual clips are bound in the `SoundLibrary` asset. **Don't `AudioSource.PlayOneShot` directly — go through `AudioManager.Instance`** so mixers and pooling stay consistent.
- **`UIManager`** (`Assets/_TakeALook/Scripts/UI/UIManager.cs`) — owns the inventory/ammo carousel panel. Two-level focus model: Level 0 = Tabs (Ammo / Inventory), Level 1 = Items inside the active tab. **The UI does not pause via `Time.timeScale` (RE4-style tactical pause)** — `FPS_Controller` and `GunSystem` poll `UIManager.IsUIPanelOpen()` to suppress player input themselves. Preserve this contract when adding new player-input systems.

### Player

- `FPS_Controller` — movement, look, jump, crouch, sprint with a stamina meter (`maxSprintStamina`, recharge + post-empty cooldown), head bob, footstep audio via `AudioManager`, interaction raycast.
- `GunSystem` — three-bullet-type weapon (`Wolf` / `Bull` / `Eagle`) configured via `GunStats` structs in the inspector. Handles draw/sheath/reload/inspect transitions, animator-driven shooting, an input-buffer window for queued shots, and the flashlight (intensity flicker via DOTween). Audio is dispatched by ID strings. Bullet types correspond to the three carousel ammo entries.
- `PlayerHealth`, `PlayerInventory` — straightforward; inventory has `StartingItem.enabled` flags for testing and is consumed by the carousel UI.

### Enemies

`EnemyAIBase` is the single AI brain (state machine: `Idle / Patrolling / Chasing / Rushing / Attacking / Fleeing / Stunned / Dead`) on a `NavMeshAgent`. Vision uses a sight-range + FOV cone with a sight-obstruction `LayerMask`. Animation events trigger projectile spawn at `shootPoint`, with safety-timeout fallbacks (`roarShootDelay`, `roarTotalDuration`, `hitStunSafetyTimeout`) so a missing event doesn't softlock the state machine. Damage is gated by `HeadshotCollider` — body shots do not damage; only the head collider (configured per enemy prefab) reports hits to `EnemyHealth`. `EnemyCorpseManager` retains corpses; `EnemyProjectile` is the enemy ranged attack.

### Items / Inventory / Carousel

- `ItemData` is an abstract `ScriptableObject` with concrete subclasses in the same file: `MedKitData`, `AmmoPackData`, `FlashlightItemData`. Each `Use(GameObject user)` returns `bool` indicating whether the use was consumed. **Add new item types by subclassing `ItemData` with a `[CreateAssetMenu]` attribute** and implementing `Use`.
- `previewMode` (`Sprite` vs `Model3D`) controls how the carousel shows the item — sprite slots use 2D hover/shake, 3D model slots use `ItemPreview3D` rotation rigs.
- `PickeableItem` writes a stable `itemId` on pickup and tells `GameManager` so it doesn't respawn.
- `CarouselUI` (`Assets/_TakeALook/Scripts/UI/Carousel/`) is generic — it's fed entries by whoever instantiates it (UIManager passes either ammo counts or inventory slots).

### Audio sound IDs

When adding audio playback, use string IDs and register the clip in the `SoundLibrary` asset. Existing categories you'll see in code: `gun_draw / gun_sheath / gun_reload / gun_inspect / gun_empty / gun_empty_reload / gun_swap`, `gun_shoot_{wolf,bull,eagle}`, `flashlight_on / flashlight_off`, `player_step`, `pickup_item`, `ui_use`. Keep the naming convention `subsystem_action[_variant]`.

### URP and quality settings

`Assets/_TakeALook/Settings/` contains the active render pipeline assets (`PC_RPAsset`, `Mobile_RPAsset`) and global volume profile. PS1 aesthetic comes from `PSXShaderKit` shaders applied via materials — don't replace those materials with URP-Lit unless the user explicitly asks for that.

## Common gotchas

- Editing prefabs in `Assets/_TakeALook/Prefabs/Enemies/` (e.g. `PF_ENEMY_CFS.prefab`, `PF_Enemy_Original 1.prefab`) often shows up as a noisy diff because Unity rewrites GUIDs of internal references. Inspect the actual semantic change before assuming the diff is real.
- The branch name has often diverged from `main` significantly (see `monedaBackUp`). Always check `git log main..HEAD` before making PRs to understand what's actually in flight.
- Spanish identifiers exist in some places (`sentivity` is a typo of `sensitivity` in `FPS_Controller`). Don't "fix" naming without checking that no inspector-bound serialized field will silently lose its value (renaming `[SerializeField]` fields without `[FormerlySerializedAs]` blanks them in every scene/prefab that referenced them).
